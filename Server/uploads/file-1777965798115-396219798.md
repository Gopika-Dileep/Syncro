## PART 1 — Permission System

### How It Works (Your Architecture)

Your system is **already well-designed**:
- Each employee has a `Permission` document with a list of `PermissionDefinition` ObjectIds
- On login, the keys (e.g. `task:create`) are loaded into the JWT / request
- `checkPermission` middleware checks if the user has the required key
- **Nothing is hardcoded** — any new role just gets a new set of keys assigned

### Answers to Your Doubts

#### ❓ Doubt 1 — Multiple Project Managers
**No changes needed.** Every PM gets the same set of keys. They all share full visibility (`project:view:all`). For example, PM_1 and PM_2 both see all company projects.

> **Your suggestion about deletion:** You're right — if a new PM takes over a project they didn't create, should they be able to delete it? The answer in a **permission-based system** is **YES, they can** — because deletion is controlled by the `project:delete` permission key, not by who created the project. The project model has no `created_by` field, so there is no ownership check. Any PM with `project:delete` can delete any project.
>
> **This is actually the correct design.** The company admin assigns who gets PM permissions. If someone is given PM permission, they are trusted with full control. You do NOT need ownership-based restrictions here. The risk is managed at the admin level (who you assign PM permissions to), not at the permission-key level.
>
> **If you ever want ownership-based deletion** (optional, for future): Add `created_by: ObjectId` to the project schema, then in the delete service, check if `req.userId === project.created_by` OR if the user has a special `project:delete:any` key. But this is overkill for now.

#### ❓ Doubt 2 — Tasks from different Team Leads under one User Story
**Fix: Add `team_id` to the Task model.** When a Lead creates a task, the backend auto-sets `task.team_id = lead's team_id`. No manual input needed.

This solves two things:
1. In Sprint Details → tasks under a user story are **grouped by team**
2. `task:view:team` works correctly — each Lead only sees tasks where `team_id` matches their team

#### ❓ Doubt 3 — Who reviews a Team Lead's assigned tasks?
Give PM the `task:review` permission (already in the set above). Lead's tasks go to IN_REVIEW → PM reviews them.

Lead's Task Page has **two slides/views**:
- **"My Team's Tasks"** (default) — uses `task:view:team`
- **"My Assigned Tasks"** — uses `task:view:assigned`, same UI as developer

#### ❓ Doubt 4 — Multiple PMs: Team Leads seeing projects from all PMs?
Works automatically. `userStory:view:all` shows all stories grouped by project name. Team Leads see the project as a label only — they never access the project detail page.

### What Data Each API Response Should Return

#### `GET /tasks/story/:storyId`
```typescript
{
  _id: string,
  title: string,
  status: string,
  priority: string,
  estimated_hours: number,
  team_id: { _id: string, name: string },    // ← ADD THIS
  created_by: { _id: string, name: string }, // ← ADD THIS (the lead who created)
  assign_to: { _id: string, name: string, avatar?: string } | null,
  rework_reason?: string,
}
```

#### `GET /tasks/team` (New endpoint for leads)
- Filters by `team_id = req.user.team_id`
- Same response shape as above

#### `GET /tasks/all` (New endpoint for PM)
- Returns ALL tasks for the company
- Same response shape including populated `team_id`, `created_by`, `assign_to`

## PART 4 — Task Status Flow (Full Reference)

```
[TODO]
  → Dev clicks "Start"  (requires: task:start)
[IN_PROGRESS]
  → Dev clicks "Submit" (requires: task:submit) → enters branch + PR link
[IN_REVIEW]
  → Lead/PM clicks "Approve" (requires: task:review) → status = Done
  → Lead/PM clicks "Redo"    (requires: task:review) → status = In Progress + rework_reason set + Rework badge shown
[DONE]
  → Final state, no further actions
```
